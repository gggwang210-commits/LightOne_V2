## 1. GitHub — 코드 저장소 활용 가이드

### 저장된 자료
- 저장소: `gggwang210-commits/LightOne_V2`
- 경로: `lightone_v2_django/` 폴더 전체 (Django MVP Rev.05)

### 언제 사용하나요?

**상황 1: 로컬에서 최신 코드를 받아야 할 때**
```bash
# 처음 받을 때 (클론)
gh repo clone gggwang210-commits/LightOne_V2

# 이미 받아둔 폴더에서 최신 코드로 업데이트할 때
cd LightOne_V2
git pull origin main
```

**상황 2: 코드를 수정하고 저장소에 올릴 때**
```bash
# 수정한 파일 확인
git status

# 변경사항 전체 추가
git add .

# 커밋 메시지 작성 (무엇을 했는지 한 줄로)
git commit -m "feat: Chart.js 대시보드 추가"

# GitHub에 업로드
git push origin main
```

**상황 3: 이전 버전으로 되돌려야 할 때**
```bash
# 커밋 이력 확인
git log --oneline

# 특정 커밋으로 되돌리기 (커밋 ID 앞 7자리 사용)
git checkout 95cbbaa -- lightone_v2_django/
```

### 커밋 메시지 작성 규칙 (권장)

| 접두어 | 의미 | 예시 |
|--------|------|------|
| `feat:` | 새 기능 추가 | `feat: Chart.js 대시보드 추가` |
| `fix:` | 버그 수정 | `fix: QS 계산 오류 수정` |
| `docs:` | 문서 수정 | `docs: README 초보자 가이드 업데이트` |
| `refactor:` | 코드 구조 개선 | `refactor: views.py 함수 분리` |

### 브랜치 전략 (향후 팀 개발 시)

```
main        ← 항상 작동하는 안정 버전 (배포용)
dev         ← 개발 중인 버전
feature/xxx ← 기능별 개발 브랜치 (예: feature/chart-dashboard)
```

## 3. Airtable — 프로젝트 관리 DB 활용 가이드

### 저장된 자료
- 베이스: `LightoneV2`
- 테이블: `트레이너 상담 리포트 SaaS`
- 저장된 레코드 6개:

| 레코드명 | 구분 | 상태 |
|---------|------|------|
| COPD Django 분석 완료 | 분석 | 완료 |
| LIGHT ONE V2 Django MVP Rev.05 완성 | 개발 | 완료 |
| 인포그래픽 Rev.5 전문가 관점 반영 완료 | 기획 | 완료 |
| LIGHTONE_V2_Complete.zip 최종 패키지 완성 | 패키징 | 완료 |
| [다음] Chart.js 변화 추세 대시보드 추가 | 개발 | 진행 예정 |
| [확인필요] 비의료 건강관리서비스 규제 해석 | 리스크 | 확인 필요 |

### 언제 사용하나요?

**상황 1: 매일 아침 오늘 할 일을 확인할 때**
1. Airtable 접속 → `LightoneV2` 베이스 열기
2. `Status` 필드가 `진행 예정`인 레코드 확인
3. 오늘 작업할 항목을 `진행 중`으로 변경

**상황 2: 작업 완료 후 기록할 때**
1. 해당 레코드의 `Status` → `완료`로 변경
2. `Notes` 필드에 완료 내용 간단히 추가
3. 새 작업이 생기면 새 레코드 추가

**상황 3: 리스크 항목을 관리할 때**
- `구분` = `리스크`인 레코드를 주기적으로 확인
- `[확인필요] 비의료 건강관리서비스 규제 해석` → 보건복지부 문의 후 결과를 Notes에 기록

### 추천 뷰(View) 설정

Airtable에서 아래 뷰를 만들면 더 편리합니다:

| 뷰 이름 | 필터 조건 | 용도 |
|---------|---------|------|
| 이번 주 할 일 | Status = 진행 예정 | 매일 확인 |
| 완료 이력 | Status = 완료 | 진행 현황 보고 |
| 리스크 관리 | 구분 = 리스크 | 위험 요소 모니터링 |

### 새 레코드 추가 방법 (Manus에게 요청 시)

"Airtable LightoneV2 베이스에 새 작업 추가해줘: [작업명], [구분], [내용]" 형식으로 요청하면 자동 저장됩니다.

## 5. 4개 서비스 연동 워크플로우

작업 한 번이 끝날 때마다 아래 순서로 진행하면 모든 서비스가 자동으로 동기화됩니다.

```
[개발 완료]
    │
    ▼
① GitHub — git add . && git commit -m "feat: xxx" && git push
    │
    ▼
② Google Drive — 변경된 문서/패키지 파일 업로드 (주요 버전 업데이트 시)
    │
    ▼
③ Airtable — 해당 레코드 Status → 완료, Notes 업데이트
    │
    ▼
④ Slack — #lightone-전체 채널에 완료 요약 메시지 전송
```

> Manus에게 "오늘 작업 내용을 4개 서비스에 저장해줘"라고 요청하면 위 과정을 자동으로 진행합니다.

## 6. 자주 묻는 질문

**Q. GitHub에 올리면 안 되는 파일이 있나요?**
A. `.env` 파일(API 키, 비밀번호 포함), `db.sqlite3`(실제 데이터), `.venv` 폴더는 올리면 안 됩니다. 프로젝트의 `.gitignore` 파일에 이미 설정되어 있습니다.

**Q. Google Drive 파일을 수정하면 자동으로 GitHub에도 반영되나요?**
A. 아닙니다. Google Drive와 GitHub는 별도로 관리됩니다. 코드는 GitHub, 문서·공유 파일은 Google Drive로 역할이 분리되어 있습니다.

**Q. Airtable 레코드가 너무 많아지면 어떻게 하나요?**
A. `완료` 상태 레코드는 별도 `아카이브` 뷰로 숨기고, 현재 진행 중인 항목만 보이도록 필터를 설정하면 됩니다.

**Q. Slack에서 Manus가 자동으로 알림을 보내게 할 수 있나요?**
A. 네. 작업 요청 시 "완료 후 Slack에 알림 보내줘"라고 함께 요청하면 자동 전송됩니다.
