# Django 배포 전 보안 및 개인정보 체크리스트

LIGHTONE V2 Django 서비스를 운영 환경에 배포하기 전에 아래 항목을 확인한다. 각 항목은 배포 승인 전 `완료` 또는 `예외 사유`를 기록해야 한다.

## 1. 운영 환경 Django 설정

- [ ] `SECRET_KEY`는 코드에 하드코딩하지 않고 운영 환경의 필수 환경변수로 주입한다.
  - 배포 스크립트, 컨테이너 런타임, PaaS 환경변수 등에서 값이 없으면 애플리케이션이 시작되지 않도록 설정한다.
  - 개발용 키와 운영용 키를 분리하고, 유출 또는 담당자 변경 시 즉시 교체한다.
- [ ] 운영 환경에서는 `DEBUG=False`를 명시한다.
  - 오류 상세 페이지, 설정값, 스택 트레이스가 외부 사용자에게 노출되지 않아야 한다.
- [ ] `ALLOWED_HOSTS`에는 실제 운영 도메인과 필요한 서브도메인만 지정한다.
  - 와일드카드(`*`) 또는 임시 개발 호스트를 운영 설정에 남기지 않는다.
- [ ] 운영 환경에서 `SESSION_COOKIE_SECURE=True`를 적용한다.
  - 세션 쿠키는 HTTPS 연결에서만 전송되도록 한다.
- [ ] 운영 환경에서 `CSRF_COOKIE_SECURE=True`를 적용한다.
  - CSRF 쿠키는 HTTPS 연결에서만 전송되도록 한다.

## 2. 저장소 커밋 금지 항목

- [ ] `.env` 파일은 Git에 커밋하지 않는다.
  - 예시 파일이 필요하면 실제 비밀값이 없는 `.env.example`만 제공한다.
- [ ] `db.sqlite3` 등 로컬 개발 데이터베이스 파일은 Git에 커밋하지 않는다.
  - 개발용 더미 데이터가 필요하면 fixture 또는 seed 스크립트로 관리한다.
- [ ] 사용자가 업로드한 미디어 원본 파일은 Git에 커밋하지 않는다.
  - 운영/파일럿 업로드 파일은 접근 권한이 통제된 별도 스토리지에 보관한다.

## 3. 개인정보 및 민감정보 처리

- [ ] 개인정보와 민감정보는 서비스 제공에 필요한 최소 범위로만 수집한다.
  - 수집 목적, 보관 기간, 접근 권한을 사전에 정의한다.
  - 주민등록번호, 건강정보, 금융정보 등 고위험 정보는 기본 수집 대상에서 제외한다.
- [ ] 개인정보 열람, 정정, 삭제 요청을 접수하고 처리하는 절차를 마련한다.
  - 요청 접수 채널, 담당자, 처리 기한, 처리 결과 통지 방식을 운영 문서에 기록한다.
  - 삭제 요청 처리 시 백업, 로그, 외부 연동 저장소의 잔존 데이터 처리 기준도 함께 확인한다.
- [ ] 운영 로그와 분석 데이터에는 개인정보와 민감정보가 불필요하게 남지 않도록 마스킹 또는 제외 처리한다.
  - 오류 추적 도구, 접근 로그, 관리자 알림 메시지에 원문 개인정보가 포함되지 않도록 점검한다.

## 4. 파일럿 데이터 관리

- [ ] 파일럿 데이터는 비식별화한 뒤 운영 데이터와 분리해 별도 보관한다.
  - 이름, 연락처, 이메일, 식별 가능한 파일명 등 직접 식별자는 제거하거나 대체한다.
  - 조합을 통해 개인이 재식별될 수 있는 항목은 범주화, 집계, 난수화 등으로 위험을 낮춘다.
- [ ] 파일럿 데이터 접근 권한은 필요한 담당자로 제한한다.
  - 접근자, 접근 목적, 반출 여부를 기록하고 파일럿 종료 후 보관 또는 삭제 여부를 결정한다.
- [ ] 외부 공유가 필요한 파일럿 자료는 공유 전 비식별화 상태와 민감정보 포함 여부를 재검토한다.
