# LIGHT ONE PT 의사결정지원 서비스 Django 구현 계획서

본 문서는 미래융합교육원 1조의 'COPD 선별 서비스' Django 프로젝트 구조를 분석하여, 이를 'LIGHT ONE 비의료 PT 상담 리포트 SaaS'로 전환 적용하기 위한 실행 계획서입니다. 의료 진단이나 처방이 아닌 트레이너의 의사결정을 보조하는 시스템(Human-in-the-loop)에 초점을 맞추어 설계되었습니다.

## 1. 프로젝트 개요 (Problem & Solution)

### 1.1 해결하고자 하는 문제
PT 현장에서는 회원의 운동 기록, 촬영 기록, 통증 반응, 상담 메모가 카카오톡, 사진첩, 메모장, 인바디 결과지 등 다양한 도구에 분산되어 있습니다. 이로 인해 트레이너는 상담 전 자료 정리에 불필요한 시간을 소모하며, 재등록 상담 시 회원에게 변화를 객관적으로 설명할 수 있는 구조화된 데이터가 부족합니다.

### 1.2 LIGHT ONE의 해결책
LIGHT ONE은 PT 현장의 데이터를 체계적으로 기록하고 분석하여 트레이너에게 제공하는 데이터 기반 PT 의사결정지원 서비스입니다. 
- **입력:** 회원 정보, 세션 기록, 운동 로그, 통증/RPE, 자세 관찰, 촬영 품질
- **처리:** 규칙 기반 점수화(QS) → 위험 라우팅(AUTO/REVIEW/BLOCK) → 트레이너 검토
- **출력:** 비의료 PT 상담 리포트, 변화 추세 대시보드

이 시스템은 트레이너를 대체하는 것이 아니라, AI가 생성한 초안을 트레이너가 검토하고 최종 판단을 내리도록 보조하는 역할을 수행합니다.

## 2. 시스템 아키텍처 (Architecture)

COPD 프로젝트의 검증된 배포 구조를 차용하여 안정적이고 효율적인 시스템을 구축합니다.

### 2.1 인프라 및 기술 스택
- **클라우드 인프라:** Google Cloud Platform (Compute Engine)
- **웹 서버:** Nginx (정적 파일 서빙, HTTPS 처리, Reverse Proxy)
- **WSGI 서버:** Gunicorn
- **백엔드 프레임워크:** Django
- **데이터베이스:** MySQL
- **프론트엔드:** HTML, CSS, JavaScript, Bootstrap
- **CI/CD:** GitHub Actions (Git Push 기반 자동 배포)

### 2.2 COPD 아키텍처 차용의 장점
- **접근 제어의 일관성:** 로그인 미들웨어를 통한 전역 보안 설정으로 권한 관리가 용이합니다.
- **역할 기반 분리:** 환자/의사 모델을 회원/트레이너 모델로 매끄럽게 전환할 수 있습니다.
- **안정적인 배포 환경:** Gunicorn 워커 최적화 및 Nginx 연동 경험을 활용하여 초기 서비스 안정성을 확보합니다.

## 3. 핵심 기능 설계 (Core Features)

### 3.1 회원 및 트레이너 관리 (Accounts)
- **역할 분리:** 사용자(User) 모델을 확장하여 회원(Member)과 트레이너(Trainer) 권한을 명확히 분리합니다.
- **보안:** `@login_required` 대신 전역 로그인 검증 미들웨어를 사용하여 모든 페이지의 접근을 제어하고, 허용된 페이지만 화이트리스트로 관리합니다.

### 3.2 세션 및 운동 기록 (Daily Care & Session)
- **일일 체크인 대체:** 회원의 세션별 상태(통증 수준, RPE, 컨디션)를 기록합니다. 중복 저장 방지 알고리즘을 적용하여 1일 1회 또는 세션당 1회로 입력을 제한합니다.
- **입력 항목:** 운동 종목, 세트, 반복 횟수, 중량, 휴식 시간, 트레이너 메모 등.

### 3.3 품질 점수(QS) 및 위험 라우팅 (Screening & Evaluation)
COPD 위험도 예측 모델의 흐름을 차용하되, 초기에는 머신러닝 대신 규칙 기반 엔진을 적용합니다.
- **QS(Quality Score) 산출:** 입력된 세션 데이터와 자세 관찰 결과를 기반으로 운동 수행 품질을 점수화합니다.
- **위험 라우팅 분류:**
  - **AUTO:** 특이사항 없음, 일반 진행 가능
  - **REVIEW:** 주의 필요, 트레이너의 면밀한 검토 요구
  - **BLOCK:** 통증이나 위험 신호 감지, 해당 동작 중단 또는 전문가 상담 권고 (의료적 판단이 아닌 안전 신호)

### 3.4 데이터 정규화 및 시각화 (Benchmarks & Dashboard)
- **상대 비율 정규화:** 서로 다른 단위의 지표(통증 점수, 자세 품질, RPE 등)를 0~100% 기준으로 정규화하여 한눈에 비교할 수 있도록 시각화합니다.
- **변화 추세 제공:** 기간별 QS 변화, 통증 감소 추세 등을 그래프로 제공하여 재등록 상담의 핵심 자료로 활용합니다.

### 3.5 비의료 PT 상담 리포트 생성 (Reporting)
- AI(규칙 엔진)가 분석한 QS 및 위험 라우팅 결과를 바탕으로 리포트 초안을 자동 생성합니다.
- 트레이너는 이 초안을 검토하고 메모를 추가하여 최종 상담 리포트를 완성합니다 (Human-in-the-loop).

## 4. 데이터베이스 모델링 (ERD)

COPD 프로젝트의 ERD를 기반으로 LIGHT ONE에 맞게 변환한 핵심 테이블 구조입니다.

| 테이블명 | 주요 필드 | 설명 |
|---|---|---|
| **User** | id, username, password, role | 회원/트레이너 구분 |
| **MemberProfile** | user_id, birth_date, sex, goals | 회원의 기본 정보 및 운동 목표 |
| **TrainerProfile** | user_id, certification_no | 트레이너 자격 및 소속 정보 |
| **SessionRecord** | member_id, date, rpe, pain_level | 세션별 회원의 주관적/객관적 상태 기록 |
| **ExerciseLog** | session_id, exercise_name, sets, reps, weight | 개별 운동 수행 기록 |
| **EvaluationResult** | session_id, qs_score, routing_status | 규칙 엔진 기반 QS 및 라우팅 결과 (AUTO/REVIEW/BLOCK) |
| **ConsultationReport** | evaluation_id, trainer_id, final_decision, memo | 트레이너가 검토 및 확정한 최종 상담 리포트 |

## 5. MVP 개발 로드맵 (Roadmap)

의료법 및 개인정보보호법 리스크를 최소화하기 위해 초기에는 가상 데이터와 더미 데이터를 활용하여 MVP를 구축합니다.

### 5.1 1단계: 기반 시스템 구축 및 데이터 수집
- Django 프로젝트 초기화 및 GCP 배포 환경 구성
- 회원/트레이너 권한 분리 및 전역 로그인 미들웨어 적용
- 세션 기록 및 운동 로그 입력 폼 개발

### 5.2 2단계: 규칙 기반 평가 및 시각화
- 규칙 기반 QS 산출 및 위험 라우팅(AUTO/REVIEW/BLOCK) 로직 구현
- 단위가 다른 건강 지표의 정규화 및 막대그래프 시각화 적용
- 트레이너용 대시보드 구축

### 5.3 3단계: 리포트 생성 및 검토 프로세스
- 트레이너 검토를 거치는 Human-in-the-loop 플로우 구현
- 최종 비의료 PT 상담 리포트(웹/PDF 형식) 생성 기능 개발

### 5.4 4단계: 고도화 (향후 계획)
- 누적된 실제 데이터를 바탕으로 XGBoost/LightGBM 기반 예측 모델 도입
- OpenPose/MediaPipe를 활용한 자세 분석 기능 추가 (품질 검증 보조용)

## 6. 리스크 관리 및 주의사항 (Risk Management)

- **의료적 표현 금지:** '진단', '치료', '처방', '재활' 등의 용어 사용을 엄격히 금지합니다. 대신 '측정', '자세 변화 추적', '운동 상담', '컨디셔닝' 등의 표현을 사용합니다.
- **안전 신호의 명확화:** BLOCK 상태는 질환에 대한 진단이 아니라, 회원의 안전을 위한 트레이닝 중단 신호임을 명확히 안내해야 합니다.
- **민감정보 처리:** `[확인필요]` 초기 MVP에서는 민감한 건강 정보 수집을 최소화하고, 향후 실제 서비스 전환 시 개인정보보호법에 따른 명시적 동의 절차를 반드시 마련해야 합니다.
